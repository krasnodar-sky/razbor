# Грабли — чек-лист типовых ловушек перед глубокой диагностикой

Каждый паттерн добыт из реальных инцидентов (майнинг журналов работы ИИ-агентов, не теория). Формат: сигнал → проверка. Пробегай список ДО того, как строить сложные гипотезы: половина «загадочных» проблем закрывается здесь.

1. **Оптимистичный интерфейс врёт.** Действие «выглядит выполненным» (клик прошёл, DOM обновился, API ответил ok), а реально не закоммитилось. → Успех подтверждай только независимым перечитыванием: перезагрузка страницы, свежий процесс, отдельный запрос к живой системе.

2. **Молчаливые сбои.** Фоновый процесс, cron или деплой «завершился успешно», а функция не выполняется неделями. → Проверяй артефакты и свежесть данных (дата изменения БД, реальные исходящие сообщения, содержимое результата), не код выхода и не зелёный статус.

3. **Зомби-состояние среды.** Осиротевшие процессы, туннели, инстансы от прошлых запусков ломают текущую задачу (два бота на одном токене, зависший headless-браузер, мёртвый VPN-интерфейс). → При загадочном сбое первым делом ищи остатки прошлых запусков; висит страница — сначала просто reload.

4. **Какая копия боевая?** Несколько машин/клонов/копий файла — компоненты смотрят в разные. → Перед выводами определи боевую копию по дате и содержимому и проверь, что все компоненты (cron, сервис, данные) смотрят на неё. Локальный клон ≠ прод.

5. **Странная цифра.** Метрика удивляет — но вывод из неё делать рано. → Сначала: как цифра физически формируется, что её обнуляет или искажает (нулевой расход при нулевом остатке = дефицит, а не отсутствие спроса)? Потом арифметика правдоподобия на пальцах.

6. **«Такого нет».** Хочется ответить «инструмента/файла/возможности не существует». → Сначала файловая или фактовая проверка (ls, find, поиск в документации), потом отрицание. Один запрос дешевле ложного «нет».

7. **Рецидив.** Проблема уже случалась и «была решена». → Прошлый фикс лечил симптом. Подними прошлый разбор и ищи, чем он был неполон; корень часто на уровень глубже (не «дубль погашен», а «CI оживляет дубль при каждом деплое»).

8. **Единственный рабочий канал.** У капризной внешней системы часто есть один задокументированный рабочий путь, а остальные стабильно виснут. → Перед дебагом проверь записи: не задокументирован ли уже рабочий канал. Не ретрай сломанный путь — иди задокументированным.

9. **Правка рвёт соседей.** «Безобидное» изменение поля/файла рушит внешние связи (переименование удаляет сопоставление в интеграции, ручная правка конфига перезаписывается живым планировщиком). → Перед правкой перечисли, кто ещё читает/синкает этот объект.

10. **Генеративный пересказ вместо источника.** LLM-извлечение и пересказы галлюцинируют — уверенно и правдоподобно. → При расхождении данных сверяйся с сырым первоисточником (сырой файл, ответ API, документ), не с интерпретацией.

11. **Секреты по содержимому, не по именам.** Перед публикацией наружу токен может жить в любом файле, не только в .env. → Скан по содержимому всех файлов, не по расширениям.

12. **Обещание вне носителя.** «Напомню», «проверим позже», недоделка в чате → потеряется. → Каждое обещание и отложенный замер — в персистентный носитель (журнал недоделок, планировщик) с датой.

## Как пополнять

Новая грабля из боевого разбора → обобщи до паттерна «сигнал → проверка» (без локальных путей, имён, ID) → допиши сюда → commit + push → `claude plugin update razbor` на машинах. Конкретику этой грабли (пути, каналы, названия систем) фиксируй в памяти своей машины.
